# Jupytext notebook workflow

Notebooks are kept in three paired forms via jupytext:

- `foo.py` — **source of truth**, written in [py:percent](https://jupytext.readthedocs.io/en/latest/formats-scripts.html#the-percent-format) format. Edit this. It's plain Python, so `mypy`, `black`, `ruff`, and `pytest` all run on it directly.
- `foo.md` — markdown rendering for human-friendly diffs (auto-generated from `.py`).
- `foo.ipynb` — Jupyter format for execution (auto-generated from `.py`). Committed without cell outputs.

**Always edit the `.py`. Never the `.md` and never the `.ipynb`.** Then sync.

## py:percent format primer

A `.py` file in py:percent looks like:

```python
# %% [markdown]
# # My notebook
#
# Some narrative — every line of a markdown cell starts with `# `.

# %%
import xarray as xr

ds = xr.open_dataset("input.nc")

# %% tags=["parameters"]
threshold = 1.0

# %%
ds.where(ds > threshold)
```

- `# %%` starts a code cell.
- `# %% [markdown]` starts a markdown cell; subsequent lines must each start with `# ` (or `#`).
- Cell tags go on the separator line: `# %% tags=["parameters"]`.

## Creating a new notebook

1. Write the `.py` file with **only** py:percent cell markers and Python content. **Do NOT hand-write any YAML frontmatter.** That means no `# ---\n# jupyter: ...\n# ---` block at the top of the file, no `kernelspec`, no `jupytext:` metadata. The frontmatter is generated by jupytext on first sync (step 2) and may be updated on later syncs — if you write it yourself you will get malformed or duplicated metadata.

   Start the file directly with a cell marker:

   ```python
   # %% [markdown]
   # # My notebook
   #
   # Some narrative.

   # %%
   import xarray as xr
   ```

2. Pair: `jupytext --set-kernel KERNEL --set-formats py:percent,md,ipynb foo.py` — this writes the YAML frontmatter as a comment block at the top of `foo.py` and creates the paired `foo.md` and `foo.ipynb`.

3. Sync (no execution): `jupytext --sync foo.py`

Check available kernels with `jupyter kernelspec list`. Common values:
`python3`, `conda-env-XXX-py`, or a custom name.

## Updating an existing notebook

Edit the `.py`, then sync code-only (the standard workflow):

```sh
jupytext --sync foo.py
```

This propagates the code from `.py` → `.md` and `.py` → `.ipynb` without running any cells. Commit all three files.

Because the `.py` is plain Python, run linters and type-checkers on it directly:

```sh
black foo.py
ruff check foo.py
mypy foo.py
```

If a tool rewrites the file (e.g. `black`), sync afterwards so `.md` and `.ipynb` stay aligned.

## Executing locally (testing only)

If you need to verify a notebook runs end-to-end before committing:

```sh
jupytext --sync --execute foo.py
jupyter nbconvert --clear-output --inplace foo.ipynb   # strip outputs
```

The clear-output step is mandatory before committing if the project keeps committed notebooks code-only. Long-running production execution should happen out-of-band (e.g. a SLURM job script that calls papermill) and write its outputs to a separate path.

Run from the notebook's directory so relative paths resolve correctly.
Prefix with the project's environment manager if needed (e.g. `pixi run`).

## Fixing a broken notebook

1. Read the `.py` to understand the code (not the `.ipynb` — it may have
   stale outputs or error markers from a prior local execution).
2. Fix the code in the `.py`.
3. Delete the `.md` and `.ipynb` and regenerate cleanly:

```sh
rm foo.md foo.ipynb
jupytext --sync foo.py            # recreates .md and .ipynb from .py, no outputs
```

Deleting the derived files first avoids stale artifacts (papermill error spans, region markers, leftover frontmatter) leaking back into the `.py` via sync.

## Cell tags and papermill

Cell tags survive the `.py` ↔ `.ipynb` ↔ `.md` roundtrip only if the `.py` is the newest of the three when you sync. If the `.ipynb` is newer (e.g. after a manual execution or a failed papermill run), jupytext syncs `.ipynb` → `.py` and may overwrite your tags. Fix: delete the derived files and regenerate from `.py`.

When using papermill for parameter injection (cluster runs, parameter sweeps), sync from `.py` first:

```sh
jupytext --sync foo.py
papermill foo.ipynb /path/to/executed/foo_params.ipynb -p threshold 2.0
```

Write papermill output to a separate path — never overwrite the source `.ipynb`, or your next `jupytext --sync` will see an `.ipynb` newer than the `.py` and pull rendered output back into the source.

## Why `.py` is the authority (not `.md` or `.ipynb`)

Three reasons:

1. **The agent can run it.** `python foo.py` executes the cells in order — useful for a fast sanity check without booting Jupyter.
2. **Standard Python tooling works.** `mypy`, `black`, `ruff`, `pytest`, `isort` all read `.py`. None of them understand `.md` notebooks or `.ipynb`.
3. **Diffs are clean.** Code review on a `.py` is just code review.

The `.md` is still committed alongside it because human-friendly markdown diffs are easier to read for narrative-heavy cells. The `.ipynb` is committed because it's what Jupyter and papermill consume directly.

## Gotchas

- **Never hand-write the `# ---\n# jupyter: ...\n# ---` block.** Jupytext writes it on first sync and updates it on later syncs. Hand-written frontmatter ends up duplicated or malformed.
- **Markdown cell continuation.** Every line of a `# %% [markdown]` cell must start with `# ` (or `#`). To produce a blank line *within* a markdown cell, use a single `#` on its own line — a fully blank line ends the cell.
- **Don't edit `.md` or `.ipynb` directly.** If something forced a change (e.g. a notebook editor saved over the `.ipynb`), either accept the new state by syncing from the modified file (`jupytext --sync` follows newest-wins) and treating the regenerated `.py` as the new source, or delete the derived files and regenerate from `.py` to discard the change.
- **`.ipynb` newer than `.py`.** Sync goes "newest wins". After papermill or a Jupyter UI save, the `.ipynb` is newer; a casual `jupytext --sync` will overwrite your `.py`. Run papermill into a separate output path to avoid this.
